# SEO Reference — Technical & On-Page for WordPress + Bricks Builder

## Technical SEO Foundations

### Crawlability & Indexing
- `robots.txt` must allow crawling of public pages and block admin/login/cart-ajax URLs
- XML sitemap generated by SEO plugin (Yoast, RankMath) — verify it includes all public pages and SureCart product URLs
- No `noindex` on pages that should rank — check meta robots tag on every template
- Canonical URLs set correctly on every page — especially important for SureCart product pages that might have multiple URL paths
- HTTPS enforced site-wide — no mixed content warnings
- Clean URL structure: `/product/screen-protector-9h/` not `/?p=123` or `/product/?id=abc`

### Core Web Vitals

| Metric | Target | What it measures |
|--------|--------|-----------------|
| LCP (Largest Contentful Paint) | < 2.5s | How fast the main content loads |
| INP (Interaction to Next Paint) | < 200ms | How fast the page responds to input |
| CLS (Cumulative Layout Shift) | < 0.1 | How much the layout shifts during load |

**LCP optimization:**
- Preload above-fold images: `fetchpriority="high"` + `loading="eager"` via Bricks `_attributes`
- Use WebP/AVIF images with appropriate sizing — don't serve a 2000px image in a 400px container
- Minimize render-blocking CSS — Bricks generates per-page CSS, which helps
- Server response time under 600ms — use caching plugin (WP Super Cache, LiteSpeed Cache, etc.)
- Preload custom fonts with `font-display: swap` (Bricks handles this when `customFontsPreload` is enabled)

**INP optimization:**
- Minimize JavaScript execution — prefer CSS for animations, hover effects, layout changes
- Defer non-critical scripts — Bricks' conditional asset loading helps (only loads element JS when element is on page)
- Avoid long tasks — break heavy JS into smaller chunks

**CLS optimization:**
- Always set explicit `width` and `height` or `aspect-ratio` on images
- Reserve space for dynamically loaded content (product grids, lazy-loaded sections)
- Web fonts: `font-display: swap` prevents invisible text but can cause layout shift — preloading the font file minimizes the swap flash
- Avoid inserting content above existing content after page load

### Image Optimization
- Use descriptive filenames: `samsung-s24-tpu-case-black.webp` not `IMG_4521.jpg`
- Serve WebP or AVIF format with JPEG/PNG fallback (use a WordPress image optimization plugin)
- Responsive images: use `srcset` and `sizes` attributes — Bricks generates these automatically for WordPress media library images
- Lazy load below-fold images (`loading="lazy"` — Bricks default)
- Eager load above-fold images (`loading="eager"` + `fetchpriority="high"`)
- Compress images: aim for under 100KB for card images, under 200KB for full-width images

### Mobile-First
- Google uses mobile-first indexing — the mobile version IS the indexed version
- All content visible on mobile without horizontal scroll
- Font sizes readable without zooming (minimum 16px for body text)
- Touch targets minimum 44x44px
- No interstitials or popups that block content on mobile

## On-Page SEO

### Page Metadata
- **Title tag**: 50-60 characters. Include primary keyword near the start. Format: `{Primary Keyword} — {Brand}` or `{Primary Keyword} | {Brand}`
- **Meta description**: 150-160 characters. Include primary keyword, a value proposition, and a call to action. This is your ad copy in search results.
- **Open Graph tags**: `og:title`, `og:description`, `og:image`, `og:url`, `og:type` — for social sharing previews
- **Twitter Card tags**: `twitter:card`, `twitter:title`, `twitter:description`, `twitter:image`

### Heading Structure
- One `<h1>` per page — the main topic/title
- `<h2>` for major sections
- `<h3>` for subsections within an `<h2>`
- Never skip levels (h1 → h3)
- Include keywords naturally — don't stuff
- Headings should make sense as a table of contents if read alone

### Content Optimization
- Primary keyword in: title, h1, first 100 words, meta description, URL slug, image alt text
- Use semantic variations and related terms — not exact-match repetition
- Content length: long enough to fully answer the searcher's intent. A product page might need 200 words. A guide might need 1500+.
- Internal links to related pages with descriptive anchor text — "9H tempered glass screen protectors" not "click here"
- External links to authoritative sources where relevant (builds trust)

### URL Structure
- Short, descriptive, keyword-rich: `/screen-protectors/` not `/category/cat-6/`
- Hyphens between words, not underscores
- Lowercase only
- No unnecessary parameters or session IDs
- Consistent trailing slash usage (pick one, stick with it)

## WordPress + Bricks Implementation

### SEO Plugin Setup
- Install Yoast SEO or RankMath — handles meta tags, sitemaps, schema basics, canonical URLs
- Configure: set homepage title/description, enable breadcrumbs, connect Search Console
- Use plugin's social settings for Open Graph defaults

### Bricks-Specific
- Bricks renders content that SEO plugins can parse for analysis
- Use the breadcrumbs element with schema output enabled
- For SureCart products: verify the SEO plugin can access `sc_product` post type for meta editing
- Don't rely on JavaScript-rendered content for SEO — critical text should be in the HTML source, not injected via JS

### Performance Plugins
- Caching: WP Super Cache, LiteSpeed Cache, or WP Fastest Cache
- Image optimization: ShortPixel, Imagify, or EWWW Image Optimizer (auto-WebP conversion)
- Consider a CDN for static assets (Cloudflare free tier is sufficient for most sites)

## Testing & Monitoring

- **Google Search Console**: monitor indexing, search performance, Core Web Vitals, mobile usability
- **Google Rich Results Test**: validate structured data on any page
- **PageSpeed Insights**: Core Web Vitals audit with specific recommendations
- **Lighthouse**: built into Chrome DevTools — SEO, Performance, Accessibility, Best Practices audits
- **Screaming Frog** (free for < 500 URLs): crawl the site for broken links, missing meta, redirect chains, duplicate content
